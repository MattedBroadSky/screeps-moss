# Screeps Moss 项目 - 需求分析文档 (v2.0)

## 1. 引言

### 1.1 目的
本文档定义了 Screeps Moss 项目的软件需求规格，为设计、开发、测试和验收提供依据。

### 1.2 范围
- **当前版本**: 任意单房间自动化管理
- **架构考虑**: 支持未来多房间扩展
- **目标用户**: 有一定编程经验的 Screeps 玩家

### 1.3 定义和缩写
- RCL: 房间控制等级
- GCL: 全局控制等级
- CPU: 游戏计算资源单位
- Tick: 游戏运行时间单位

## 2. 总体描述

### 2.1 产品前景
构建模块化、可扩展的 Screeps AI 系统，实现从单房间到多房间的全周期自动化管理。

### 2.2 用户特征
- 主要用户: 有编程经验的 Screeps 玩家
- 技术能力: 熟悉 JavaScript，能进行基本配置
- 使用目标: 自动化游戏管理，专注策略制定

### 2.3 约束
#### 技术约束
- CPU: 每 tick 20-100（根据 RCL）
- 内存: 2MB 限制
- API: 游戏提供的有限 API
- 实时性: 必须在 tick 时限内完成

#### 项目约束
- 时间: 3个月核心功能，6个月稳定版
- 资源: 1名主要开发者（AI辅助）
- 质量: 99.9% uptime，严格性能标准

### 2.4 假设和依赖
#### 技术假设
- Screeps API 保持向后兼容
- JavaScript 运行时稳定
- 网络连接基本稳定
- 游戏服务器可用性 > 99%

#### 用户假设
- 用户有基本编程知识
- 用户愿意配置系统
- 用户会提供反馈

#### 外部依赖
- Screeps 游戏服务器
- GitHub 代码仓库
- 社区文档和支持

## 3. 具体需求

### 3.1 功能性需求

#### FR-001: 能量管理系统 (P0)
**描述**: 自动管理能量采集、存储、分配
**输入**: 房间能量状态、Source信息、存储设施状态
**输出**: 采集分配方案、存储填充指令、监控数据
**验收标准**:
1. 能量中断30分钟内恢复
2. 能量使用效率 > 80%
3. 存储填充率60-90%
4. 无能量浪费

#### FR-002: Creep生成系统 (P0)
**描述**: 自动生成、替换、管理Creep
**输入**: 房间状态、当前Creep、可用能量
**输出**: 生成决策、身体配置、角色分配
**验收标准**:
1. 关键角色始终足够
2. 死亡前100tick内生成替代
3. 身体配置优化
4. 内存管理良好

#### FR-003: 角色行为系统 (P0)
**描述**: 为Creep分配和执行任务
**输入**: Creep角色状态、可用任务、房间布局
**输出**: 任务分配、行为指令、效率报告
**验收标准**:
1. 任务完成率 > 95%
2. 无任务冲突
3. 移动效率优化
4. API调用优化

#### FR-004: 建筑管理系统 (P1)
**描述**: 自动规划、建造、维护建筑
**输入**: 房间布局、建筑状态、可用资源
**输出**: 布局规划、建造任务、维护计划
**验收标准**:
1. 布局合理支持扩展
2. 按优先级建造
3. 损坏24小时内修复
4. RCL升级进度符合预期

#### FR-005: 内存管理系统 (P1)
**描述**: 管理游戏内存，防止泄漏
**输入**: 当前内存使用、数据状态
**输出**: 清理操作、状态持久化、监控告警
**验收标准**:
1. 内存使用 < 1.5MB
2. 无内存泄漏
3. 关键状态持久化
4. 定期清理无效数据

#### FR-006: 状态监控系统 (P1)
**描述**: 实时监控游戏状态
**输入**: 游戏各项指标
**输出**: 状态报告、趋势分析
**验收标准**:
1. 实时显示关键指标
2. 历史数据可查询
3. 状态变化及时检测
4. 监控覆盖率 > 90%

#### FR-007: 性能监控系统 (P2)
**描述**: 监控系统性能指标
**输入**: CPU使用、内存使用、响应时间
**输出**: 性能报告、优化建议
**验收标准**:
1. CPU使用率监控
2. 内存使用监控
3. 响应时间监控
4. 性能告警及时

#### FR-008: 异常告警系统 (P2)
**描述**: 检测和告警异常情况
**输入**: 系统状态、错误信息
**输出**: 告警通知、处理建议
**验收标准**:
1. 关键异常5分钟内告警
2. 告警信息明确
3. 处理建议合理
4. 告警历史可追溯

#### FR-009: 日志记录系统 (P2)
**描述**: 记录系统操作和事件
**输入**: 操作数据、事件信息
**输出**: 结构化日志、分析报告
**验收标准**:
1. 关键操作有日志
2. 日志结构化可分析
3. 日志存储安全
4. 日志查询方便

#### FR-010: 自动化部署系统 (P3)
**描述**: 一键部署到游戏
**输入**: 代码变更、配置更新
**输出**: 部署结果、状态报告
**验收标准**:
1. 部署成功率 > 95%
2. 部署时间 < 5分钟
3. 回滚机制有效
4. 部署日志完整

### 3.2 非功能性需求

#### 扩展性需求

### 3.3 接口需求

#### 内部接口
- 模块间通过清晰API通信
- 数据格式统一（JSON）
- 错误处理机制一致
- 日志记录标准统一

#### 外部接口
- Screeps游戏API
- 部署脚本接口
- 监控系统接口
- 配置管理接口

### 3.4 数据需求

#### 数据结构
```javascript
// 房间状态
RoomState = {
  name: string,
  controllerLevel: number,
  energy: {available: number, capacity: number},
  creeps: Array<CreepInfo>,
  structures: Array<StructureInfo>
}

// Creep配置
CreepConfig = {
  role: string,
  body: Array<string>,
  priority: number,
  minCount: number,
  maxCount: number
}

// 建筑布局
BuildingLayout = {
  type: string,
  position: {x: number, y: number},
  priority: number,
  status: string
}
```

#### 数据存储
- 实时数据: Game.memory (2MB)
- 配置数据: JSON文件
- 历史数据: 本地备份
- 监控数据: 时间序列

### 4.2 详细用例

#### 用例1: 管理房间能量
- **参与者**: 玩家/系统
- **前置条件**: 房间已占领
- **主流程**: 检测状态 → 分配采集 → 管理存储 → 监控平衡
- **后置条件**: 能量稳定供应
- **异常**: 中断时紧急恢复

#### 用例2: 生成管理Creep
- **参与者**: 系统
- **前置条件**: 有可用能量
- **主流程**: 分析需求 → 生成决策 → 分配角色 → 生命周期管理
- **后置条件**: Creep数量满足需求
- **异常**: 能量不足时优化配置

#### 用例3: 监控系统状态
- **参与者**: 玩家/系统
- **前置条件**: 系统运行中
- **主流程**: 收集指标 → 分析状态 → 显示报告 → 异常告警
- **后置条件**: 状态可视，问题可追溯
- **异常**: 监控故障时降级运行
