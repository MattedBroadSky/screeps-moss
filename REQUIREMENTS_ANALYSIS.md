## 🎯 1. 项目概述

### 1.1 项目背景
Screeps 是一款面向程序员的 MMO 策略游戏，玩家通过编写 JavaScript 代码控制游戏单位。本项目旨在构建一个自我进化、模块化、可扩展的 AI 系统。

### 1.2 项目目标
- **短期目标**: 恢复游戏运行，建立稳定基础
- **中期目标**: 实现自动化管理，优化资源利用
- **长期目标**: 构建智能决策系统，支持多房间扩展

### 1.3 项目范围
- **包含**:
  - 单房间自动化管理 (E15S17)
  - 模块化代码架构
  - 实时监控系统
  - 自动化部署流水线
- **不包含**:
  - 多房间协同（初期）
  - PvP 战斗策略（初期）
  - 市场交易系统（初期）

## 🔍 2. 通用现状分析

### 2.1 典型 Screeps 项目起点
```
典型起点状态:
- 房间: 新占领的房间 (RCL 1)
- 能量: 300 (Spawn 初始能量)
- Creep: 0
- 基础设施: 1 Spawn
- 资源: 2个 Source，1个 Mineral
```

### 2.2 通用关键问题
1. **能量管理**: 建立稳定的能量采集、存储、分配系统
2. **Creep 生命周期**: 生成、替换、回收的完整管理
3. **建筑规划**: 从 RCL 1 到 RCL 8 的渐进式建筑规划
4. **防御系统**: 入侵检测和自动防御
5. **性能优化**: 在 CPU 和内存限制下的高效运行

### 2.3 通用技术约束
- **CPU 限制**: 每 tick 有限计算资源（通常 20-100 CPU）
- **内存限制**: 2MB 内存空间
- **API 限制**: 游戏提供的有限 API 接口
- **代码体积**: 影响加载时间和内存使用
- **实时性要求**: 每 tick 必须在时限内完成计算

## 🏗️ 4. 系统架构需求

### 4.1 架构原则
1. **模块化设计**: 功能解耦，独立开发
2. **分层架构**: 表现层、业务层、数据层分离
3. **插件化扩展**: 功能可插拔，易于扩展
4. **配置驱动**: 行为由配置控制，非硬编码

### 4.2 技术栈需求
- **开发语言**: JavaScript (Screeps 环境)
- **构建工具**: 自定义构建脚本
- **版本控制**: Git + GitHub
- **部署工具**: 自定义部署脚本
- **监控工具**: OpenClaw + 自定义监控

### 4.3 数据存储需求
- **内存存储**: Game.memory, Memory 对象
- **持久化存储**: 本地文件系统 (状态备份)
- **配置存储**: JSON 配置文件
